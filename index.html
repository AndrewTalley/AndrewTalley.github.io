5 microservices in prod

Five Node.js microservices and a Java file-generation application make up the platform. Here are the four pieces of work I'd walk through in an interview.

Led the migration of a legacy .NET 4.5 application to .NET 8.0, moving it off on-prem servers and onto Kubernetes. Replaced manual SFTP delivery with AWS S3 orchestrated through Control-M, and added structured logging and metrics so the team could see problems before they became outages.

.NET 4.5 → 8.0 On-prem → Kubernetes SFTP → S3 + Control-M

I designed and own the team's ETL pipeline - a chain of services that turns market config in MongoDB into 60–80 delivered files a day. The scheduler (cron) reads config from Mongo and kicks off datagenerator, which pulls the per-market queries and runs them; dataformatter renders the raw results as CSV, XML, or DSV; and the Java filegenerator writes each file and ships it over SFTP. evv-dbutil wraps all the Mongo access.

60–80 files / day CSV · XML · DSV SFTP delivery

Rebuilt the team's internal job-execution tool from vanilla JavaScript to React, so roughly ten business users could trigger and monitor jobs themselves instead of filing a ticket. Production-support load dropped by about 30%. Later built an automated reporting pipeline that queries MongoDB and uses AI to generate charts and dashboards, after taking over reporting ownership from another team.

~10 self-serve users −30% support load AI-generated reporting

Stood up Snowflake for the team and rewrote 6 to 8 existing MSSQL queries as Snowflake views across three markets, consolidating classic and cloud data into one source of truth and eliminating duplicate pulls.

Flagship project · full-stack esports platform

PCC - Prime Championship Circuit

A competitive-league platform for the MOBA Predecessor - months of work spanning a React/TypeScript front end and a Go (Gin + GORM) REST + WebSocket API. It pulled match and rank data from the Omeda City API and wrapped it in the tooling a competitive scene actually needs: profiles, teams, scheduling, tournaments, and a server-authoritative real-time draft room.

  • Server-authoritative draft room - WebSocket ban/pick with captain validation, server-run timers, per-team time banks, bonus time, and side selection.
  • Swiss-system tournaments - automated pairings with rematch avoidance and byes, live brackets, and admin-gated creation.
  • Automated stat sync - player and team profiles backed by a worker that polled the Omeda City API every few minutes.
  • Steam + email auth - Steam OpenID and email/password login with confirmation, JWT access/refresh, and role-gated admin routes.
  • Moderated uploads - team logos stored in S3 and auto-screened for explicit content with AWS Rekognition.
  • Live social layer - WebSocket notifications, a social feed, news and patch notes, plus a companion Discord bot.

Recent project · Predecessor analytics PWA

Ortus.gg

The spiritual successor to PCC - a Predecessor companion and analytics site rebuilt on the game's new GraphQL API (the one that replaced Omeda City). It's a single self-contained Go binary: Gin and templ render the UI server-side, a WebSocket engine drives a real-time draft tool, and a build analyzer turns raw match data into hero insights.

  • Real-time draft tool - WebSocket drafts with multiple modes, server-run timers, spectator links, plus draft history, export, and per-draft stats.
  • Build analyzer - turns raw match data into hero build insights and matchup analysis.
  • Hero database - searchable heroes with detail pages for abilities and perks, rendered server-side with templ.
  • Tier list - ranked heroes computed from match data.
  • GraphQL data layer - typed queries against the Predecessor GraphQL API.
  • Installable PWA - service worker and manifest, mobile-tuned, with static assets embedded into the Go binary and deployed on Railway.
